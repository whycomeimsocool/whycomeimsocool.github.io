Sound in Paper

Privacy-first, QR-based audio sharing — put sound into paper

Pre-launch · All milestones code-complete Next.js 15 · TypeScript · Postgres · Prisma · Web Crypto API · MinIO/R2 · Stripe

A musician hands someone a small printed card. They scan it. A private song plays. No app to install, no account to create, no social profile to navigate. The QR code is not a link to a streaming service — it is the listening object itself. Sound in Paper is built around the premise that the physical artifact is the product, and the service exists to put sound into paper.

The privacy model is architectural. Audio, artwork, title, and message are encrypted in the browser before upload using the Web Crypto API with AES-GCM. The server stores only encrypted objects. The decryption key lives in the URL fragment — never sent over HTTP — so the server can serve a Piece without being able to hear it. A printed QR can be updated to play different audio at any time without reprinting; the paper stays the same.

All ten milestones are code-complete: Auth.js v5 with PBKDF2 key derivation and IndexedDB non-extractable trusted-device keys; encrypted audio upload to S3-compatible object storage; dynamic QR generation with stable fragment share keys; browser-side decryption with one-tap playback; six listener themes; privacy-friendly analytics with no raw IP retention; abuse reporting and DMCA intake with admin moderation UI; expiration, play limits, and Pass-gated listening; Stripe billing and plan enforcement; and printable PDF, card, and sticker layouts for physical QR artifacts. 86 unit tests pass. 32 of 32 Playwright acceptance tests pass against local MinIO.

Custom CMS & Admin Panel

Full-stack content management system built from scratch

Live Python · SQLite · PHP · vanilla JS · ES modules

A bespoke content management system built entirely from scratch for a public educational website. The public-facing site is intentionally lightweight — PHP, JSON, and vanilla JavaScript — while the local admin panel is a sophisticated browser UI backed by a Python standard-library HTTP server with modular handlers, a dual-mode SQLite/JSON storage adapter, and an extensive suite of utility scripts.

The system manages posts, action artwork, resources, QR code generation and scan analytics, access log ingestion and geolocation, a Wikipedia tracker with stale-while-revalidate caching, and publishing workflows that sync SQLite state to public JSON. Post ingestion alone handles URL normalization, tracking-parameter removal, platform detection, duplicate checking across active and archived records, thumbnail extraction, and article archiving. The architecture was designed for a single operator running the admin panel locally and publishing to a remote PHP host — a constraint that shaped every design decision.

The codebase includes Python unit tests via pytest, frontend tests via Jest, and documentation covering current architecture, admin workflows, operations and maintenance, and a full configuration reference.

AstroCal

Astronomical calendar with lunar, solar, and timezone precision

Live JavaScript · Astronomy Engine · Hebcal · SVG · CSS

What began as a Word table — a more beautiful calendar than the printable ones available online — became a study in astronomical precision. The calendar renders moon phases, lunar and solar eclipses, hemisphere-aware iconography, full timezone support with UTC conversion, Dawn and Midnight Zmanim via the Hebcal library, and an interactive legend. All phase and eclipse data is calculated using Don Cross's Astronomy Engine, chosen specifically for the accuracy of its mathematics.

All symbols are Unicode icons converted to SVGs, hand-aligned to the pixel. The interface evolved iteratively: navigation, then hemispheres, then eclipses, then timezones, then Zmanim — each addition integrated into the existing architecture rather than bolted on. A native app version of the calendar extends this work to mobile.

Astronomical Calendar — iOS & Android App

React Native adaptation of AstroCal for native mobile

In development React Native · TypeScript · Zustand · Astronomy Engine · Hebcal

A cross-platform mobile app adapting the web calendar's calculation engine into a native iOS and Android experience. The app coordinates multiple independent calculation domains — astronomical rise/set events, photography light windows, moon phases, eclipse detection, and halachic Zmanim — running them in parallel with fault-tolerant merging so a failure in one domain never degrades the others.

The architecture separates concerns cleanly: moonDataService owns all phase, eclipse, illumination, and age data for the visible month in a single pass; timeService orchestrates parallel astronomy and halachic fetches; geocodingService handles GPS, manual entry, Nominatim search, and reverse geocoding. A small Zustand store holds only what is genuinely shared across screens. The web calendar's original calculation logic is preserved in a web_reference/ directory and the React Native code migrates intent while replacing DOM patterns with native navigation, gesture paging, and platform font registration.

Musical Interval Typography System

A type scale derived from the notes of a musical scale

Live · CC BY-NC 4.0 HTML · CSS custom properties · zero dependencies

Born from a single question: can the intervals of a musical scale, rather than arbitrary numerical values, define the sizes of type? The answer is a pure CSS system — no JavaScript, no build step, no dependencies — in which a single ratio (the minor third, expressed as 32/27) governs every measurement on the page: font sizes, margins, padding, line heights, and vertical rhythm.

The system is fully fluid and responsive with practically no media queries. It uses clamp() and CSS custom properties to scale from the minimum accessible font size upward, and a parallel vw-unit track handles body and margin proportions independently. The code is published openly on the site with a full written explanation of how it works and why each variable exists. The result is what the site you are reading now is built on.

EduToneBox

Real-time educational synthesizer · C++ · Qt6 · Raspberry Pi

In development C++17 · Qt6/QML · CoreAudio · ALSA · FFTW3 · OSC · CMake

A physical educational synthesizer prototype: a C++17 DSP engine connected over OSC/UDP to a Qt6/QML control surface, deployable on both macOS and Raspberry Pi. The goal is to make synthesis concepts visible and playable — waveform shape, frequency, amplitude, ADSR envelopes, LFO modulation, multi-algorithm filtering, and effects — with real-time FFT spectrum visualization feeding back to the UI.

The audio engine and UI are deliberately separate processes. DSP modules — oscillator with PolyBLEP band-limiting, ADSR, LFO, state variable filter, Butterworth, virtual-analog, and modal resonator algorithms, reverb, delay, chorus, flanger, distortion, and bit crusher — are plain C++ classes with explicit state, independently unit-tested via Catch2 and CTest. The Raspberry Pi build path adds ALSA, FFTW3, and ARM/NEON optimization hooks with full deploy and audio-test scripts. The OSC control API is fully documented.

33 of 34 CTest tests pass. The single known failure is a test that predates PolyBLEP implementation and expects hard-edge waveform samples — the fix is already understood, not yet written. The roadmap includes MIDI input, preset storage, hardware encoder mapping, and guided lesson modes.
