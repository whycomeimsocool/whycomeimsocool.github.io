Sound in Paper

Put sound into paper — a privacy-first, QR-based audio sharing system for artists and anyone who wants to share sound as an intimate object

Next.js 15 · TypeScript · Postgres · Prisma · Web Crypto API · Cloudflare R2

In development  ·  Milestones 0, 1, and 2 complete

The Idea

A musician hands someone a small printed card. They scan it. A private song plays. No app to install, no account to create, no social profile to navigate. Just sound, unlocked by paper.

Sound in Paper is built around the premise that the physical artifact is the product — not the platform behind it. The QR code is not a link to a streaming profile. It is the listening object itself. The creator feels like they are preparing a piece of paper that contains sound, not publishing content to a cloud service.

Control the paper = control the song.

A printed QR code can be left in a gallery, slipped inside a record sleeve, handed to someone at a show, or placed anywhere in the physical world. The creator can update the sound behind it at any time without reprinting. The same paper continues to work.

What It Should Feel Like

It should feel like

  • a message passed by hand
  • a small art object
  • a secret song container
  • a minimalist music player
  • a physical-world sound object

It should not feel like

  • a promo landing page
  • a streaming smart link
  • a social profile
  • a CRM or marketing dashboard
  • a generic QR generator

Privacy Architecture

The privacy model is architectural, not merely a policy. The server hosts encrypted sound objects and is deliberately designed so it cannot listen to them.

Client-side encryption

Audio, artwork, title, and message are encrypted in the browser before upload using the Web Crypto API. The server stores only encrypted files, encrypted metadata, non-sensitive routing metadata, and aggregate event records. It never receives plaintext audio, titles, messages, or artwork.

URL fragment key

The QR code points to a URL shaped like: 

https://soundinpaper.com/p/quiet-river-17#k=PUBLIC_SHARE_KEY

The slug is sent to the server. The fragment — the share key after # — is never sent in normal HTTP requests. The browser uses it locally to decrypt the Piece.

This means the server can serve the encrypted manifest for quiet-river-17 without ever knowing the key needed to decrypt it. The decryption capability lives only in the printed QR code and in the creator's encrypted key set.

Dynamic QR

A printed QR code should keep working even when the creator changes the audio behind it. Each Piece has a stable public share key embedded in the QR. New versions are encrypted so that the same printed paper can decrypt the current assigned content. The creator's encrypted key set retains the Piece key so updates remain possible.

Key management

On account creation, a creator master key is generated in the browser. A recovery phrase — shown once — can unlock it. Only encrypted key material is stored server-side. If a creator loses their recovery phrase and has no trusted device remaining, the company cannot recover their encrypted content. This is explained during setup in plain language, without making the product feel like a security bunker.

The Listener Experience

The listener page shows almost nothing by default: a large play button, an optional progress bar, an optional title, an optional short message, and an optional artwork surface. No creator identity, no promotional links, no social mechanics, no engagement bait.

A quiet response control — present like a small folded corner of paper — can reveal lightweight actions such as a heart and a report. It is secondary to listening in every sense: visually, spatially, and philosophically.

Playback pipeline

  • Listener browser downloads the encrypted audio blob
  • Web Crypto decrypts it locally using the share key from the URL fragment
  • Browser creates a temporary object URL from the decrypted bytes
  • HTML audio element plays the object URL
  • No plaintext audio ever transits the network

Moderation and Safety

The privacy model creates a real tension with safety. The product does not pretend otherwise. The moderation principle is: no surveillance, clear safety boundaries, human review only when someone with access reports abuse.

When a listener files a report, the report flow explains that submitting it shares the Piece key with moderation — making this an explicit, informed action rather than a silent surveillance mechanism. Moderators can decrypt reported Pieces for review and disable Piece routes. DMCA notice and counter-notice intake is built into the milestone plan, along with repeat-infringer tracking.

Product Guardrails

  • Do not turn this into a link-in-bio product
  • Do not add external music service links in the MVP
  • Do not upload plaintext audio to the server
  • Do not store plaintext titles, messages, artwork, or audio
  • Do not use magic-link-only login
  • Do not add ads or social mechanics to listener pages
  • Do not promise guaranteed autoplay
  • Do not add per-upload copyright checkboxes
